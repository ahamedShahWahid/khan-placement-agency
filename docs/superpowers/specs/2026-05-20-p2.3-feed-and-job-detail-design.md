# P2.3 ‚Äî Feed + job detail endpoints ‚Äî design

**Date:** 2026-05-20
**Spec ref:** `IMPLEMENTATION_SPEC.md` ¬ß10 (API surface ‚Äî `/v1/feed`, `/v1/jobs/{id}`, cursor pagination, ETag, RFC 7807), ¬ß6.3 (matching surfaces above threshold), ¬ß13 P2 ("feed endpoint"). Reference designs: P2.2 matches table (`docs/superpowers/specs/2026-05-20-p2.2-matches-and-scoring-design.md`).
**Status:** Draft ‚Äî pending approval.

Fourth P2 slice. P2.0‚ÄìP2.2 built the data: jobs are embedded, applicants are embedded, matches are computed and surfaced. This slice exposes that data through two read endpoints ‚Äî the applicant feed and a single-job detail view. The endpoints are the first thing the Flutter web app (F1) reads.

## Why this slice

- **Flutter F1 cannot start without these endpoints** ‚Äî the feed UI consumes `/v1/feed`, the job detail screen consumes `/v1/jobs/{id}`.
- **The schema is already shaped for this** ‚Äî `matches.surfaced_at` exists for the surfaced filter, the partial-DESC index `ix_matches_applicant_surfaced` exists for the order-and-cursor path, the `jobs(status, posted_at DESC)` index covers any job-only lookups.
- **No external dependencies.** This slice is purely SQL queries + cursor encoding + response serialization. Fits cleanly inside the existing FastAPI surface.

## Decisions resolved during brainstorming

| # | Decision | Resolution |
|---|---|---|
| 1 | Feed card content | **Rich card** ‚Äî job + employer + match (total + vector + structured + components). Lets P2.4 render "why this fits" without a follow-up request. Lean / two-shape rejected ‚Äî round-trip cost and client-side resolution complexity don't pay off at MVP scale. |
| 2 | Cursor encoding | **Opaque base64 of `{score, match_id}` JSON.** Decode to `(Decimal, UUID)`, query as `WHERE (total_score, id) < (cursor_score, cursor_id)` for stable cursor across ties. Pure decoding ‚Äî no server state, no expiry, replayable. |
| 3 | Order | `ORDER BY matches.total_score DESC, matches.id DESC` ‚Äî id tiebreaker keeps the cursor unambiguous when scores tie. |
| 4 | Page size | Default 20, max 50, configurable via `?limit=N`. Per spec ¬ß10 ("cursor-based"). |
| 5 | ETag | **Weak ETag** `W/"<sha256(applicant_id + max(updated_at in page) + count)>"`. Strong ETag would require byte-exact reproducibility we don't guarantee. `If-None-Match` ‚Üí 304 with no body. |
| 6 | Single-job 404 | **Uniform** ‚Äî unknown id AND `status != 'open'` AND `deleted_at IS NOT NULL` all return `404 job_not_found`. Distinguishing would leak existence (same rationale as `/v1/applicants/me/resumes/{id}` per CLAUDE.md). |
| 7 | Match on job detail | **Returned if any match row exists for `(current_applicant, job)`**, regardless of `surfaced_at`. An applicant pasting a URL sees the score even if the match didn't make their feed. |
| 8 | Applicant gate | **`_require_applicant` reused** from `routes/resumes.py`. Recruiter/admin tokens ‚Üí `403 not_an_applicant` before any DB read. |
| 9 | Closed-job feed behavior | **Already filtered by `surfaced_at IS NOT NULL` + `status = 'open'`** in the query. If a job closes after a match was surfaced, that match drops from the feed. Predictable. |
| 10 | Pagination cursor on tie | **stable** because the ORDER BY includes `id DESC`. Two matches with `total_score = 0.75` are returned in deterministic id-DESC order across pages. |

## Endpoints

### `GET /v1/feed`

**Auth:** Bearer JWT ‚Üí `current_user` ‚Üí `_require_applicant` ‚Üí applicant row.

**Query parameters:**

- `limit: int = 20` (clamp to `[1, 50]`).
- `cursor: str | None = None` (base64-encoded JSON `{"score": "0.8500", "match_id": "uuid"}`).

**Response 200 (`application/json`):**

```json
{
  "items": [
    {
      "match": {
        "id": "uuid",
        "total_score": 0.8500,
        "vector_score": 0.9200,
        "structured_score": 0.7400,
        "components": {"location": 1.0, "exp": 1.0, "ctc": 0.5},
        "surfaced_at": "2026-05-20T12:34:56+00:00"
      },
      "job": {
        "id": "uuid",
        "title": "Senior Backend Engineer",
        "description": "...",
        "locations": ["Bangalore", "Remote"],
        "min_exp_years": 5,
        "max_exp_years": 9,
        "ctc_min": 2500000.00,
        "ctc_max": 4200000.00,
        "posted_at": "2026-05-15T09:00:00+00:00"
      },
      "employer": {
        "id": "uuid",
        "name": "Acme Co",
        "verified": true
      }
    }
  ],
  "next_cursor": "eyJzY29yZSI6IjAuNzUwMCIsIm1hdGNoX2lkIjoiLi4uIn0="
}
```

`next_cursor` is `null` on the final page. Always present in the response shape ‚Äî clients shouldn't conditionally check for the key.

**Headers:**

- `ETag: W/"<sha256-hex>"` on 200.
- `If-None-Match` echoed in request ‚Üí `304 Not Modified` with empty body.

**Errors (problem+json):**

- `401 missing_bearer_token` / `401 invalid_access_token` / `401 user_not_found` ‚Äî from `current_user`.
- `403 not_an_applicant` ‚Äî from `_require_applicant`.
- `400 invalid_cursor` ‚Äî base64 decode failed or JSON missing required keys or `match_id` not a uuid.
- `400 limit_out_of_range` ‚Äî `limit < 1` or `limit > 50`.

**Query:**

```sql
SELECT m.*, j.*, e.id, e.name, e.verified_at
FROM kpa.matches m
JOIN kpa.jobs j ON j.id = m.job_id AND j.deleted_at IS NULL AND j.status = 'open'
JOIN kpa.employers e ON e.id = j.employer_id AND e.deleted_at IS NULL
WHERE m.applicant_id = :applicant_id
  AND m.deleted_at IS NULL
  AND m.surfaced_at IS NOT NULL
  AND (:cursor_score IS NULL OR (m.total_score, m.id) < (:cursor_score, :cursor_id))
ORDER BY m.total_score DESC, m.id DESC
LIMIT :limit + 1
```

`LIMIT :limit + 1` ‚Äî peek at one extra row to decide whether `next_cursor` should be set. Trim before serialization.

`ix_matches_applicant_surfaced (applicant_id, total_score DESC) WHERE deleted_at IS NULL AND surfaced_at IS NOT NULL` is the index Postgres uses. The `(m.total_score, m.id) < (cursor_score, cursor_id)` tuple comparison maps cleanly onto the index for cursor seeks.

### `GET /v1/jobs/{job_id}`

**Auth:** same as `/v1/feed`.

**Path:** `job_id` must parse as UUID. Malformed ‚Üí `400 invalid_uuid` (FastAPI's default UUID path-param parsing handles this; converted to RFC 7807 via the existing error handler).

**Response 200:**

```json
{
  "job": { /* same shape as feed.job */ },
  "employer": { /* same shape as feed.employer */ },
  "match": { /* same shape as feed.match ‚Äî or null if no match row exists */ }
}
```

**Headers:** ETag `W/"<sha256(job.id + job.updated_at + match.updated_at?)>"`.

**Errors:**

- `401`, `403` ‚Äî same as feed.
- `404 job_not_found` ‚Äî uniform across "unknown id", "soft-deleted job", "status=closed".
- `400 invalid_uuid`.

**Query:**

Two queries (cleaner than a LEFT JOIN against optional match):

```sql
-- Job + employer
SELECT j.*, e.id, e.name, e.verified_at
FROM kpa.jobs j
JOIN kpa.employers e ON e.id = j.employer_id AND e.deleted_at IS NULL
WHERE j.id = :job_id AND j.deleted_at IS NULL AND j.status = 'open'

-- Match (optional)
SELECT m.*
FROM kpa.matches m
WHERE m.applicant_id = :applicant_id AND m.job_id = :job_id AND m.deleted_at IS NULL
```

The match query may return zero rows; the response carries `match: null` in that case.

## Pydantic `*Read` models

In `routes/feed.py`:

```python
from pydantic import BaseModel, ConfigDict


class MatchRead(BaseModel):
    model_config = ConfigDict(from_attributes=True)

    id: uuid.UUID
    total_score: float
    vector_score: float
    structured_score: float
    components: dict[str, float]
    surfaced_at: datetime | None


class EmployerRead(BaseModel):
    model_config = ConfigDict(from_attributes=True)

    id: uuid.UUID
    name: str
    verified: bool                       # derived from verified_at IS NOT NULL


class JobRead(BaseModel):
    model_config = ConfigDict(from_attributes=True)

    id: uuid.UUID
    title: str
    description: str
    locations: list[str]
    min_exp_years: int
    max_exp_years: int
    ctc_min: float | None
    ctc_max: float | None
    posted_at: datetime


class FeedItemRead(BaseModel):
    match: MatchRead
    job: JobRead
    employer: EmployerRead


class FeedResponse(BaseModel):
    items: list[FeedItemRead]
    next_cursor: str | None


class JobDetailResponse(BaseModel):
    job: JobRead
    employer: EmployerRead
    match: MatchRead | None
```

`MatchRead.components` reads from the JSONB column as a `dict[str, float]`. `EmployerRead.verified` is derived in the route from `Employer.verified_at IS NOT NULL` ‚Äî the wire shape exposes a bool, not a timestamp (timestamp is internal accounting).

## Cursor helpers (in `routes/feed.py`)

```python
import base64
import json
from decimal import Decimal
from uuid import UUID


def encode_cursor(score: Decimal, match_id: UUID) -> str:
    payload = {"score": str(score), "match_id": str(match_id)}
    return base64.urlsafe_b64encode(json.dumps(payload).encode("utf-8")).decode("ascii")


def decode_cursor(cursor: str) -> tuple[Decimal, UUID]:
    """Returns (score, match_id). Raises ValueError on any malformed input."""
    try:
        raw = base64.urlsafe_b64decode(cursor.encode("ascii"))
        payload = json.loads(raw)
        return Decimal(payload["score"]), UUID(payload["match_id"])
    except (ValueError, KeyError, TypeError, json.JSONDecodeError, binascii.Error) as exc:
        raise ValueError(f"invalid_cursor: {exc}") from exc
```

Route handler catches `ValueError` ‚Üí HTTPException with `400 invalid_cursor`.

## ETag helper (in `routes/feed.py`)

```python
def make_weak_etag(*parts: object) -> str:
    """W/"<sha256-hex>" of the str-rendered parts joined by '|'."""
    raw = "|".join(str(p) for p in parts)
    return f'W/"{hashlib.sha256(raw.encode("utf-8")).hexdigest()}"'
```

## What's out of scope (intentional)

- **Recruiter "candidates for this job" endpoint** ‚Äî `ix_matches_job_surfaced` exists but the HTTP surface is deferred.
- **`PATCH /v1/me`** for applicant profile updates that would trigger rescore ‚Äî separate slice.
- **Match explanation text** (P2.4 ‚Äî blocked on BRD ¬ß14 #1).
- **Save / apply endpoints** (P3.0).
- **Server-side caching (Redis)** for the feed ‚Äî spec ¬ß10 mentions it; deferred until load tests justify.
- **OpenAPI schema customization beyond what FastAPI auto-generates** ‚Äî Flutter client codegen consumes the auto-generated schema as-is in this slice.

## Risks

- **Cursor leaks scoring boundary.** A cursor with `score=0.5499` reveals that the requesting client is paginating just below threshold. Mitigation: this is the applicant's own data; the cursor is keyed to their session via the auth token. Not a security issue at MVP scope; revisit if the feed becomes public.
- **ETag has false positives on JSONB churn.** If `score_components` re-serializes with a different dict order, the SQL-level `max(updated_at)` doesn't change but the body bytes do. Mitigation: weak ETag (`W/`) signals "semantically equivalent", which is the right semantic here.
- **Pagination + concurrent rescore.** If a `score_applicant` rescore changes total_score between page 1 and page 2 of the same client session, the cursor positions may shift ‚Äî a match could appear twice or disappear. Mitigation: `id DESC` tiebreaker prevents the double-show; missed-match is rare in practice (rescores are infrequent post-MVP). Accept for MVP; add `version` field in the cursor for full snapshot consistency if data shows the issue.
- **`description` field is large** (job descriptions are 100‚Äì500 words). 20-item page √ó 500 words ‚âà 200KB per response. Mitigation: at MVP scale acceptable; if it becomes a problem, ship a `?summary=true` variant that drops `description`.
