# P2.0 — Jobs table + seeding — design

**Date:** 2026-05-20
**Spec ref:** `IMPLEMENTATION_SPEC.md` §5 (data model — `employers`, `jobs`), §13 P2 ("Recruiter direct posting, job embedding, hybrid scoring, feed endpoint" — this slice picks off the foundational table only).
**Status:** Draft — pending approval.

First P2 slice. Recruiter-side endpoints are deliberately deferred (applicant-side + Flutter web is the agreed P2/P3 scope). This slice gets `employers` and `jobs` rows into the database via a versioned JSON fixture loaded by a CLI script, so that the next two slices (P2.1 job embeddings, P2.2 scoring) have data to operate on, and so a backend-only demo of the future feed is possible.

## Why this slice

- **P2.1 and P2.2 both need rows in `jobs`.** Job embeddings encode job text; matching scores `(applicant_embedding, job_embedding)` pairs. Neither can be written, let alone tested end-to-end, without the table.
- **Recruiter HTTP CRUD is out of scope for this iteration of P2/P3.** Without it, "create a job" needs an alternative path. A seed CLI fed by a versioned JSON fixture is the smallest credible substitute and is reusable (P2.2 matches seeding, P3.1 notification template seeding will follow the same pattern).
- **The data model deserves to land in isolation.** Bundling employers + jobs with embeddings or matching in one slice would mix three concerns (schema, async pipeline, scoring algorithm) and obscure review.

## Decisions resolved during brainstorming

| # | Decision | Resolution |
|---|---|---|
| 1 | Employer dependency | `employers` table lands in this slice alongside `jobs`. Free-text `employer_name` and JSON-embedded snapshot rejected as schema churn. |
| 2 | Job status enum | Two states: `open`, `closed`. `draft` / `paused` / `expired` deferred until recruiter UI lands — adding them later is a single `ALTER TYPE` migration. |
| 3 | Seeding mechanism | Standalone CLI (`python -m kpa.scripts.seed_jobs`) reading a versioned JSON fixture at `api/data/sample_jobs.json`. Alembic data migrations rejected (mixes schema with data); pytest-only rejected (blocks backend demo). |
| 4 | Demo dataset realism | Realistic Indian-context dataset: ~10 employers (mixed verticals), ~27 jobs (varied seniority, INR salaries, Indian + Remote locations). Faker-generated rejected as non-reproducible for future matching evaluation. |
| 5 | Employer idempotency key | Stored `name_norm` column (lowercased + whitespace-collapsed), partial-UNIQUE indexed where `deleted_at IS NULL`. Generated columns rejected for partial-index lookup cost. |
| 6 | Job idempotency key | Script-level only — SELECT by `(employer_id, lower(title))`, no DB constraint. Real recruiters re-list roles with identical titles; the DB must not block that. |
| 7 | Job FK semantics | `employer_id ... ON DELETE RESTRICT`. Employers are soft-deleted; a hard delete that orphans jobs is a deliberate decision, not a silent cascade. |
| 8 | Experience-years column type | `INTEGER` on jobs (`min_exp_years`, `max_exp_years`), keeps `NUMERIC(4,1)` on `applicants.years_experience`. Asymmetric on purpose — job posts say "3–5 years", applicants are "4.5 years". P2.2 scoring handles the cast. |
| 9 | JSON fixture date encoding | `posted_days_ago: int` not `posted_at: timestamp`. Loader converts to `now() - timedelta(days=...)` so the checked-in dataset doesn't visibly age. |
| 10 | Pydantic model location | Co-located in `seed_jobs.py`, not a separate `schemas.py`. ~40 lines, one consumer. |

## Surface

This slice ships no HTTP endpoints. Three new things land:

**1. Two new SQLAlchemy models** in `db/models.py` (sketch — full DDL in the migration):

```python
class JobStatus(StrEnum):
    OPEN = "open"
    CLOSED = "closed"


class Employer(Base):
    __tablename__ = "employers"

    id: Mapped[UuidPK]
    name: Mapped[str] = mapped_column(String(200), nullable=False)
    name_norm: Mapped[str] = mapped_column(String(200), nullable=False)
    gst: Mapped[str | None] = mapped_column(String(15), nullable=True)
    verified_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True), nullable=True)
    created_at: Mapped[CreatedAt]
    updated_at: Mapped[UpdatedAt]
    deleted_at: Mapped[DeletedAt]

    __table_args__ = (
        Index(
            "ix_employers_name_norm_live",
            "name_norm",
            unique=True,
            postgresql_where="deleted_at IS NULL",
        ),
        {"schema": "kpa"},
    )


class Job(Base):
    __tablename__ = "jobs"

    id: Mapped[UuidPK]
    employer_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True),
        ForeignKey("kpa.employers.id", ondelete="RESTRICT"),
        nullable=False,
    )
    title: Mapped[str] = mapped_column(String(200), nullable=False)
    description: Mapped[str] = mapped_column(Text, nullable=False)
    locations: Mapped[list[str]] = mapped_column(
        ARRAY(String(100)), nullable=False, server_default="{}"
    )
    min_exp_years: Mapped[int] = mapped_column(Integer, nullable=False)
    max_exp_years: Mapped[int] = mapped_column(Integer, nullable=False)
    ctc_min: Mapped[float | None] = mapped_column(Numeric(12, 2), nullable=True)
    ctc_max: Mapped[float | None] = mapped_column(Numeric(12, 2), nullable=True)
    status: Mapped[JobStatus] = mapped_column(
        SAEnum(
            JobStatus,
            name="job_status",
            native_enum=True,
            schema="kpa",
            values_callable=lambda x: [e.value for e in x],
        ),
        nullable=False,
        default=JobStatus.OPEN,
        server_default=JobStatus.OPEN.value,
    )
    posted_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now(), nullable=False
    )
    created_at: Mapped[CreatedAt]
    updated_at: Mapped[UpdatedAt]
    deleted_at: Mapped[DeletedAt]

    __table_args__ = (
        Index(
            "ix_jobs_employer_id_live",
            "employer_id",
            postgresql_where="deleted_at IS NULL",
        ),
        Index(
            "ix_jobs_status_posted_at_live",
            "status",
            text("posted_at DESC"),
            postgresql_where="deleted_at IS NULL",
        ),
        CheckConstraint(
            "max_exp_years >= min_exp_years",
            name="ck_jobs_exp_years_ordered",
        ),
        CheckConstraint(
            "ctc_max IS NULL OR ctc_min IS NULL OR ctc_max >= ctc_min",
            name="ck_jobs_ctc_ordered",
        ),
        {"schema": "kpa"},
    )
```

The `(status, posted_at DESC)` partial index is the feed-query path called out in spec §5 — defining it now means P2.3's `/v1/feed` doesn't have to revisit the migration.

**2. Migration 0005** (`db/migrations/versions/0005_employers_and_jobs.py`):

- `CREATE TYPE kpa.job_status AS ENUM ('open', 'closed')`.
- `CREATE TABLE kpa.employers (...)` with partial-unique index on `name_norm`.
- `CREATE TABLE kpa.jobs (...)` with two partial indexes and two CHECK constraints.
- Hand-written (consistent with repo norm — autogenerate is off until ~10 tables exist).
- `downgrade()` drops in reverse order; jobs first (FK), then employers, then the enum.

**3. Seed CLI** (`src/kpa/scripts/seed_jobs.py`) — entry shape:

```python
def main() -> int:
    args = _parse_args()
    payload = _load_and_validate(args.path)        # Pydantic validation
    report = asyncio.run(_apply(payload, dry_run=args.dry_run))
    _log.info("seed.complete", **report.as_log_kwargs())
    return 0
```

The async core `_apply()` opens one session against the same engine the API uses, runs employer upserts then job upserts in dependency order, commits once (or rolls back on `--dry-run`).

Pyproject `[project.scripts]` adds a `kpa-seed-jobs` shorthand so `uv run kpa-seed-jobs --dry-run` works alongside the long form.

## Data flow

```
sample_jobs.json
      │
      ▼
_load_and_validate()  ──── Pydantic: SeedPayload(version, employers[], jobs[])
      │                    cross-validates job.employer_name ∈ employers
      ▼
_apply()              ──── one async session, one transaction
      │
      ├── for each employer:
      │     SELECT id FROM employers WHERE name_norm = ? AND deleted_at IS NULL
      │     ├── if found: UPDATE gst, verified_at  (name, name_norm preserved)
      │     └── else:     INSERT
      │
      ├── for each job:
      │     SELECT id FROM jobs
      │     WHERE employer_id = ? AND lower(title) = lower(?) AND deleted_at IS NULL
      │     ├── if found: UPDATE description, locations, min/max exp, ctc, status, posted_at
      │     └── else:     INSERT
      │
      └── COMMIT (or ROLLBACK on --dry-run)
```

`updated_at` flips automatically via the model's `onupdate=func.now()`. `posted_at` is recomputed from `posted_days_ago` on each run — re-seeding "ages forward" past dates, which is the desired behavior for an evergreen demo fixture.

## JSON fixture shape (v1)

```json
{
  "version": 1,
  "employers": [
    {
      "name": "Acme Fintech Pvt Ltd",
      "gst": "27AABCU9603R1Z2",
      "verified": true
    }
  ],
  "jobs": [
    {
      "employer_name": "Acme Fintech Pvt Ltd",
      "title": "Senior Backend Engineer",
      "description": "We're hiring a backend engineer ...",
      "locations": ["Bangalore", "Remote"],
      "min_exp_years": 5,
      "max_exp_years": 8,
      "ctc_min": 2500000,
      "ctc_max": 4200000,
      "status": "open",
      "posted_days_ago": 3
    }
  ]
}
```

Field-by-field validation rules:

- `version` must equal `1`. Future shapes bump.
- `employers[*].name` length 1–200 after strip.
- `employers[*].gst` optional; if present, length-15 string (no checksum check — recruiter-side concern).
- `employers[*].verified` bool; maps to `verified_at = now() if true else NULL` on insert. On update, `true` overwrites an existing `NULL` but does not overwrite an existing non-NULL `verified_at` (preserve the original verification time).
- `jobs[*].employer_name` must match an entry in `employers[]` exactly (post-strip, case-sensitive — the fixture authors mistakes here are typos, not whitespace, and a strict match catches the typos).
- `jobs[*].title` length 1–200.
- `jobs[*].description` non-empty.
- `jobs[*].locations` list of length 0–10 of 1–100-char strings.
- `jobs[*].min_exp_years`, `max_exp_years` 0–50 integers; `max_exp_years >= min_exp_years`.
- `jobs[*].ctc_min`, `ctc_max` optional decimals; if both present, `ctc_max >= ctc_min`. Either may be `null`.
- `jobs[*].status` `"open"` or `"closed"`. Default `"open"`.
- `jobs[*].posted_days_ago` non-negative int, ≤ 3650 (10y sanity cap).

## CLI surface

```
$ uv run python -m kpa.scripts.seed_jobs --help
usage: seed_jobs [-h] [--from PATH] [--dry-run]

Seed employers and jobs from a JSON fixture (idempotent).

options:
  --from PATH     Path to seed JSON. Default: api/data/sample_jobs.json
  --dry-run       Parse + validate the JSON, log what would change, do not write.
```

Exit codes: `0` success, `2` JSON or validation error, `3` DB error. (`1` reserved for argparse usage errors.)

Logging via `structlog.get_logger(__name__)`, matching the rest of the codebase. Key events:

- `seed.start path=... dry_run=...`
- `seed.employer.inserted name="..." id=...`
- `seed.employer.updated name="..." id=...`
- `seed.job.inserted employer="..." title="..." id=...`
- `seed.job.updated employer="..." title="..." id=...`
- `seed.complete employers_inserted=N employers_updated=M jobs_inserted=N jobs_updated=M dry_run=...`

## Dataset characteristics

The shipped `api/data/sample_jobs.json` contains:

- **10 employers:** 4 tech product, 2 fintech, 2 edtech, 1 services, 1 manufacturing. 7 of 10 `verified: true`. GST strings present for the 7 verified.
- **27 jobs:** 6 entry (0–2y), 12 mid (3–6y), 9 senior (7–12y). CTC ranges in lakhs INR (₹4L–₹50L). Every job 1–3 locations from {Mumbai, Bangalore, Hyderabad, Pune, Delhi NCR, Chennai, Remote}. `posted_days_ago` spread 0–45. All `status: "open"` (closed-status coverage handled in unit tests, not the fixture).
- **Skill diversity in `description`:** backend, frontend, full-stack, data, DevOps, ML, product, design, QA — spread to exercise vector-similarity variance in P2.1.

The fixture is the canonical demo dataset; the integration test `test_loader_against_sample_jobs_json` re-asserts the counts so drift is caught at PR time.

**Not tested in P2.0:**

- Argparse / CLI entrypoint plumbing — thin shim; testing it would require subprocess + real DB.
- Concurrent seed runs — dev-loop tool only. If we ever need it, `INSERT ... ON CONFLICT` is the natural fix.
- Cross-table validation (`applicant_embeddings`, `matches`) — tables don't exist yet; coverage belongs in P2.1+.

## What's out of scope (intentional)

- **Recruiter HTTP CRUD** (`POST /v1/employers/{eid}/jobs` etc.). Deferred to a future recruiter-side slice outside the current P2/P3 applicant-only scope.
- **Job embeddings** (`job_embeddings` table, `embed_job` Celery task). P2.1.
- **Matching** (`matches` table, score worker, hybrid algorithm). P2.2.
- **Feed endpoint** (`/v1/feed`). P2.3.
- **Match explanations** (LLM provider interface, templated fallback). P2.4. Blocked on BRD §14 Open Decision #1.
- **`draft`, `paused`, `expired` job statuses.** Add when recruiter UI needs them.
- **Job expiry beat task.** Needs `expired` status to exist first.
- **GSTIN checksum validation.** Recruiter-side concern.
- **Per-employer logo / branding columns.** Visual polish, not foundational. Add when frontend asks.

## Risks

- **Seed fixture goes stale.** `posted_days_ago` mitigates the obvious "all jobs posted in 2026" problem, but salary bands and titles will drift from market reality. Mitigation: a comment at the top of `sample_jobs.json` notes "regenerate annually" — and the data is small enough to refresh in a single PR.
- **Hand-rolled SELECT-then-UPSERT race.** Two concurrent `seed_jobs` runs could double-insert. Accepted — this is a developer tool, not a request path. If a load-test scenario ever needs concurrent seeding, swap to `INSERT ... ON CONFLICT (name_norm) WHERE deleted_at IS NULL DO UPDATE`.
- **CHECK constraints surface as Postgres errors, not validation errors.** A malformed JSON that passes Pydantic but fails the CHECK constraint (e.g., a `min_exp_years: -1` slips through if we forget the Pydantic bound) returns exit code 3 with an opaque psycopg error. Mitigation: Pydantic bounds are intentionally tighter than CHECKs so the validation layer catches it first; the CHECK is defense in depth for direct SQL.
- **Engine instantiation pulls in `settings.py`.** A misconfigured `.env` makes the seeder fail with the same `Settings` validation message the API would. Acceptable — single source of truth for env config.
