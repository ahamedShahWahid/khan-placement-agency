# P2.1 — Job embedding worker — design

**Date:** 2026-05-20
**Spec ref:** `IMPLEMENTATION_SPEC.md` §5 (data model — `job_embeddings`), §6.3 (matching uses cosine over job + applicant vectors), §13 P2 ("job embedding"). Reference design: `docs/superpowers/specs/2026-05-19-embedding-worker-design.md` (the applicant-side embedding worker shipped in P1.3).
**Status:** Draft — pending approval.

Second P2 slice. P2.0 landed `employers` + `jobs` + a seed CLI; rows exist in the database but carry no embeddings. P2.2's matching pipeline cannot run until applicant and job vectors live in the same space. This slice ships the job side of that space, dispatched from the seed CLI per upsert.

## Why this slice

- **P2.2 matching is blocked.** §6.3 cosines `applicant_embedding` against `job_embedding`. With only applicant vectors today, the join is unwriteable.
- **The provider, queue, and worker shape are already in place** from P1.3. This slice is largely a "same machine, second input" exercise — but the canonicalization differs (no parse step) and the dispatch trigger differs (the seed CLI, not a parse worker's Txn 3).
- **Bundling embedding with matching would mix three concerns** (canonicalization, async pipeline, scoring algorithm). Keep them separate.

## Decisions resolved during brainstorming

| # | Decision | Resolution |
|---|---|---|
| 1 | Dispatch trigger | **Seed CLI dispatches per inserted/updated job** post-commit, fire-and-forget, broad-except + `_log.warning("embed.dispatch-failed", ...)`. Mirrors the parse-resume → embed-applicant pattern. No backfill CLI in this slice — re-running the seed CLI re-dispatches because the seeder is idempotent end-to-end. |
| 2 | Canonical text fields | **Title + description + employer name + sorted locations + experience band.** CTC and status excluded. CTC is noisy numeric signal that doesn't embed well; status is always `open` for embedded rows; both are scored at the structured layer (P2.2). |
| 3 | Canonical text format | One line `title: {title} at {employer} \| text: {description}` + optional `Locations: ...` line + `Experience: ...` line. Hash determinism by sorting locations, normalizing line endings, dropping empty `Locations:`. |
| 4 | Per-entity cardinality | **One current vector per job.** `job_embeddings.job_id UNIQUE`. UPSERT via `pg_insert(...).on_conflict_do_update(...)` — same shape as `applicant_embeddings`. |
| 5 | FK semantics | `job_embeddings.job_id ... ON DELETE CASCADE`. Soft-deletes of jobs leave the embedding (deleted_at filtering at query time); only hard-deletes ever fire the cascade. Mirrors `applicant_embeddings`. |
| 6 | Behavior on `open → closed` | Embedding stays. Matching filters on `Job.status='open'` at query time. Cheaper than re-embedding on every close/re-open. |
| 7 | Idempotency gate | **`canonicalized_text_hash` column on `job_embeddings`.** Txn 1 computes the hash and exits early if it matches the existing row — no provider call, no row write. |
| 8 | Model identification | `model_name` column on the row (not implicit from config) — a future model rotation can target rows where `model_name != current_default` for backfill. |
| 9 | Backfill CLI | Deferred. The seed CLI's idempotent re-run gives free re-dispatch. If broker-down recovery becomes a real operational concern, a `kpa-embed-jobs --missing` CLI is a one-task add. |
| 10 | Re-embedding under a new model | Out of scope. When the model rotates, a one-shot mass UPSERT against all jobs solves it; doesn't need infrastructure today. |

## Surface

Three new things ship plus modifications to two existing modules.

**1. New SQLAlchemy model** in `db/models.py`:

```python
class JobEmbedding(Base):
    __tablename__ = "job_embeddings"

    id: Mapped[UuidPK]
    job_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True),
        ForeignKey("kpa.jobs.id", ondelete="CASCADE"),
        nullable=False,
        unique=True,
    )
    embedding: Mapped[list[float]] = mapped_column(Vector(1536), nullable=False)
    model_name: Mapped[str] = mapped_column(String(64), nullable=False)
    canonicalized_text_hash: Mapped[str] = mapped_column(CHAR(64), nullable=False)
    input_tokens: Mapped[int] = mapped_column(Integer, nullable=False)
    created_at: Mapped[CreatedAt]
    updated_at: Mapped[UpdatedAt]
    deleted_at: Mapped[DeletedAt]
```

Identical shape to `ApplicantEmbedding` — the only swap is `applicant_id` → `job_id`.

**2. Migration 0006** (`db/migrations/versions/0006_job_embeddings.py`):

- `CREATE TABLE kpa.job_embeddings` with `unique(job_id)`, vector(1536), the soft-delete trio.
- `CREATE INDEX ix_job_embeddings_hnsw ON kpa.job_embeddings USING hnsw (embedding vector_cosine_ops)` via `op.execute()` (the same form 0004 uses for the applicant HNSW index).
- No `CREATE EXTENSION vector` — 0004 already added it. Including a defensive `IF NOT EXISTS` form is optional and a no-op.
- `downgrade()`: drop HNSW index → drop table. Don't drop the `vector` extension — `applicant_embeddings` still depends on it.

**3. Canonicalizer** in `api/src/kpa/integrations/embeddings/canonicalize_job.py`:

```python
def canonicalize_job(job: Job, *, employer_name: str) -> tuple[str, str]:
    title = job.title.strip()
    emp = employer_name.strip()
    desc = job.description.strip().replace("\r\n", "\n").replace("\r", "\n")
    locs = sorted(
        (loc.strip() for loc in (job.locations or []) if loc.strip()),
        key=str.lower,
    )
    if job.min_exp_years == job.max_exp_years:
        exp = f"{job.min_exp_years} years"
    else:
        exp = f"{job.min_exp_years}-{job.max_exp_years} years"

    lines = [f"title: {title} at {emp} | text: {desc}"]
    if locs:
        lines.append(f"Locations: {', '.join(locs)}")
    lines.append(f"Experience: {exp}")

    text = "\n".join(lines)
    hash_hex = hashlib.sha256(text.encode("utf-8")).hexdigest()
    return text, hash_hex
```

Separate file from `canonicalize.py` (the applicant canonicalizer). Different input shapes, no shared internals, and the applicant module is already at a comfortable size — splitting keeps each focused.

**4. Worker** in `api/src/kpa/workers/tasks/embed_job.py`:

Same decorator stack and same `_run_async` thread-hop helper as `embed_applicant`. The task body is:

```python
@celery_app.task(
    name="kpa.embed_job",
    bind=True,
    max_retries=3,
    autoretry_for=(TransientEmbeddingError,),
    retry_backoff=2,
    retry_backoff_max=60,
    retry_jitter=True,
    acks_late=True,
)
def embed_job(self, job_id_str: str) -> None:
    _run_async(_embed_job_async(uuid.UUID(job_id_str)))
```

Three transactions:

- **Txn 1 (gate):** Load `Job` joined with `Employer.name`. If either is soft-deleted, log `embed_job.skipped` and return. Compute `(text, hash_hex)`. Look up existing `JobEmbedding` by `job_id`. If `existing.canonicalized_text_hash == hash_hex`, log `embed_job.no-op` and return.
- **Txn 2 (Gemini, no DB):** `provider.encode(text=text, task=EmbeddingTask.DOCUMENT, title=f"{job.title} at {employer.name}")`. Returns `EmbeddingResult(vector, model_name, input_tokens)`. May take a few seconds — no DB connection held.
- **Txn 3 (re-verify + UPSERT):** Re-read job + employer in a fresh session, recompute `hash_now`. If `hash_now != hash_hex`, the seed CLI or future recruiter CRUD updated the job mid-flight — log `embed_job.aborted reason=hash-drifted` and return (the next re-seed re-dispatches). Otherwise UPSERT via `pg_insert(...).on_conflict_do_update(...)` on `job_id`.

**5. Seed CLI dispatch:**

`_apply_in_session` is extended so it records the job IDs it touched on the report:

```python
@dataclass
class SeedReport:
    employers_inserted: int = 0
    employers_updated: int = 0
    jobs_inserted: int = 0
    jobs_updated: int = 0
    inserted_job_ids: list[uuid.UUID] = field(default_factory=list)
    updated_job_ids: list[uuid.UUID] = field(default_factory=list)
    dry_run: bool = False
```

`_upsert_job` appends to the right list. The outer `_apply` calls `_dispatch_embeds(report.inserted_job_ids + report.updated_job_ids)` **after the COMMIT** (skipped under `--dry-run`):

```python
def _dispatch_embeds(job_ids: list[uuid.UUID]) -> None:
    for jid in job_ids:
        try:
            embed_job.delay(str(jid))
        except Exception:                            # broad on purpose — broker down ≠ seed failure
            _log.warning("embed.dispatch-failed", job_id=str(jid), exc_info=True)
```

`as_log_kwargs()` deliberately omits the ID lists — per-row log events already exist; the summary line stays clean.

## Data flow

```
seed CLI run
    │
    ▼
_apply_in_session  ──  upsert employers + jobs (collect inserted_/updated_job_ids)
    │
    ▼
COMMIT
    │
    ▼
_dispatch_embeds(job_ids)
    │       ├── embed_job.delay(jid₁)   ◀── broker down? log + continue
    │       ├── embed_job.delay(jid₂)
    │       └── ...
    ▼
seed.complete

         ──── (asynchronous; Celery worker on -Q embed) ────

embed_job(jid)
    │
    ▼  Txn 1: load Job+Employer, canonicalize, compare hash
    │
    │       ├── deleted? → log skipped, return
    │       └── hash matches existing? → log no-op, return
    ▼
provider.encode(...)          ◀── transient err? autoretry up to 3
    │
    ▼  Txn 3: re-read, re-canonicalize, verify hash unchanged
    │
    │       └── hash drifted? → log aborted, return
    ▼
pg_insert(JobEmbedding).on_conflict_do_update(job_id) → COMMIT
```

## Important divergences from P1.3 (intentional)

- **No parse worker upstream.** Job text is structured-ish from upsert time. `canonicalize_job` reads the `Job` row directly + an `Employer.name` join.
- **Dispatch source is the seed CLI**, not a worker's Txn 3. Wrapping discipline is the same (broad except + `_log.warning("embed.dispatch-failed", ...)`), but the call site moves.
- **`canonicalize_job` lives in its own file** (`integrations/embeddings/canonicalize_job.py`). The applicant canonicalizer in `canonicalize.py` is unchanged.
- **No `embed_status` column on `Job`.** Same reasoning as applicant — the embedding either exists or doesn't, no intermediate state surfaced to users.

**Test infrastructure deltas:**

- `tests/integration/conftest.py`: add `kpa.job_embeddings` to the `concurrent_async_client` TRUNCATE list, before `kpa.jobs`.
- `tests/integration/conftest.py`: extend `patched_embedding_provider` to patch `kpa.workers.tasks.embed_job.get_embedding_provider` and seed the cache for the embed_job side too.

**What's deliberately not tested in P2.1:**

- **HNSW recall/latency.** P2.2 owns matching quality.
- **Concurrent embed dispatches for the same job.** `on_conflict_do_update` makes the race benign (last write wins; both writes carry identical content if input hadn't changed).
- **Re-embedding under a different model_name.** Out of scope until model rotation is operationally needed.

## What's out of scope (intentional)

- **Recruiter HTTP CRUD that dispatches `embed_job`.** Deferred; the seed CLI is the only dispatch site in this slice.
- **Backfill CLI** (`kpa-embed-jobs --missing`). Deferred — re-seed handles broker-outage recovery for now.
- **The `matches` table + scoring** (P2.2).
- **`/v1/feed` and `/v1/jobs/{id}` endpoints** (P2.3).
- **Match explanations / LLM provider interface** (P2.4 — blocked on BRD §14 Open Decision #1).
- **Model rotation tooling.**

## Risks

- **Hash drift race in Txn 3.** The seeder updating a job mid-flight could surface as `embed_job.aborted reason=hash-drifted`. Mitigation: structurally benign — the next re-seed re-dispatches and a fresh embed completes. The integration test `test_embed_job_aborts_on_hash_drift_in_txn3` exercises this path, closing the P1.3 retro's gap.
- **Embedding text format drift.** A future change to `canonicalize_job` would invalidate every stored hash, forcing a full re-embed. Mitigation: the `model_name` column gives a fingerprint to filter on, and `test_canonical_format_basic` is a snapshot guard against accidental drift.
- **Gemini cost on full re-seed.** 27 jobs is ~negligible (a few cents). A future fixture with thousands of jobs would notice. Mitigation: idempotency by hash already protects re-seeds that don't actually change content.
- **`gemini-embedding-2` model EOL.** Tracked at the application-config layer (`KPA_EMBEDDING_MODEL`). When Google deprecates, set `KPA_EMBEDDING_MODEL=<new>` and run a backfill. The `model_name` column makes the backfill targetable.
