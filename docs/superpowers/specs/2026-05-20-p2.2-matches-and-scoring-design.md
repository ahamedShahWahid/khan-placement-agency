# P2.2 — Matches table + hybrid scoring — design

**Date:** 2026-05-20
**Spec ref:** `IMPLEMENTATION_SPEC.md` §5 (data model — `matches`), §6.3 (matching: hybrid scoring + threshold), §13 P2 ("hybrid scoring"). Reference designs: `docs/superpowers/specs/2026-05-19-embedding-worker-design.md` (P1.3 applicant embedding) and `docs/superpowers/specs/2026-05-20-p2.1-job-embedding-worker-design.md` (P2.1 job embedding).
**Status:** Draft — pending approval.

Third P2 slice. P2.0 + P2.1 placed applicants and jobs into the same embedding space; this slice ships the score computation that joins them, the `matches` table that stores per-pair scores, and the two Celery workers that materialize matches when either side gets a new embedding. P2.3 turns the surfaced subset into a feed.

## Why this slice

- **The feed (P2.3) reads from `matches`.** No matches → no feed → no demo of the core product loop.
- **Scoring is on the critical path for the BRD's "first match in 10 min" target.** The structured budget (parse 8s + embed 1s + score initial 10 jobs 4s = 13s) only works if scoring runs without external API calls — pure CPU over already-stored vectors.
- **The algorithm has knobs the BRD calls out as needing calibration** (weights, threshold, per-rule weights). Landing them as env-driven settings with reasonable defaults lets us tune without code changes after we ship labeled data.

## Decisions resolved during brainstorming

| # | Decision | Resolution |
|---|---|---|
| 1 | Structured score richness | **Lean: location + experience band + CTC overlap** — the 3 components whose inputs already exist on the schema. `must_have_skills[]` and `dealbreakers[]` deferred to a small follow-up that also extends the seed JSON. Skipping the structured score entirely was rejected — the spec's stated failure mode is "noisy matching" and pure-vector is exactly that. |
| 2 | Score-dispatch trigger(s) | **Both `score_applicant` and `score_job`.** Dispatched from the respective embed worker's Txn 3 post-commit, fire-and-forget, broad-except + `_log.warning("score.dispatch-failed", ...)`. Mirrors the parse→embed and seed→embed wiring. |
| 3 | Cosine implementation | **Python-side**, not pgvector's `<=>` operator. Score worker iterates the full open-jobs / all-applicants list (not a top-K), so HNSW isn't on the path. P2.3 feed will use HNSW-backed `ORDER BY <=> LIMIT N` for top-K. `cosine_similarity` lives in `kpa/scoring/vector.py` behind a single import so the swap is localized. |
| 4 | Match cardinality | **One row per `(applicant_id, job_id)` live pair.** UPSERT on rescore via partial-UNIQUE `WHERE deleted_at IS NULL`. Soft-delete a match → next rescore inserts a fresh row. |
| 5 | Below-threshold behavior | **Store the row with `surfaced_at IS NULL`**, don't drop it. Keeps recall data for offline eval; feed filters on `surfaced_at IS NOT NULL`. |
| 6 | `surfaced_at` on rescore | **Preserve, never unset.** Once a match crosses threshold, it stays surfaced even if a later rescore drops below. Predictable for users; aligns with "match was relevant at some point" semantics. |
| 7 | Per-rule structured weights | **Equal weights (1/3 each)** as the unweighted mean. The vector-vs-structured top-level split is `0.6 / 0.4` per spec §6.3, exposed as `KPA_MATCH_VECTOR_WEIGHT`. Per-rule promotion to a config table is deferred. |
| 8 | Threshold | **Configurable via `KPA_MATCH_SURFACE_THRESHOLD`**, default `0.55` (matches spec). |
| 9 | Stored metadata on each match | **JSONB columns `score_components` and `model_versions`.** Per-rule breakdown for debugging + which embedding models / weights produced the row. Cheaper than schema versioning. |
| 10 | Match `ON DELETE` behavior | **CASCADE on both FKs.** Matches without their applicant or job are meaningless. Soft-deletes don't cascade (handled by query-time `deleted_at IS NULL` filter). |

## Surface

Three pure-Python scoring modules, two Celery tasks, one new table, two new env vars.

### `matches` table (migration 0007)

```python
class Match(Base):
    __tablename__ = "matches"

    id: Mapped[UuidPK]
    applicant_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True),
        ForeignKey("kpa.applicants.id", ondelete="CASCADE"),
        nullable=False,
    )
    job_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True),
        ForeignKey("kpa.jobs.id", ondelete="CASCADE"),
        nullable=False,
    )
    vector_score: Mapped[float] = mapped_column(Numeric(5, 4), nullable=False)
    structured_score: Mapped[float] = mapped_column(Numeric(5, 4), nullable=False)
    total_score: Mapped[float] = mapped_column(Numeric(5, 4), nullable=False)
    score_components: Mapped[dict[str, Any]] = mapped_column(JSONB, nullable=False)
    model_versions: Mapped[dict[str, Any]] = mapped_column(JSONB, nullable=False)
    surfaced_at: Mapped[datetime | None] = mapped_column(
        DateTime(timezone=True), nullable=True
    )
    created_at: Mapped[CreatedAt]
    updated_at: Mapped[UpdatedAt]
    deleted_at: Mapped[DeletedAt]

    __table_args__ = (
        Index(
            "ix_matches_applicant_job_live",
            "applicant_id",
            "job_id",
            unique=True,
            postgresql_where="deleted_at IS NULL",
        ),
        Index(
            "ix_matches_applicant_surfaced",
            "applicant_id",
            text("total_score DESC"),
            postgresql_where="deleted_at IS NULL AND surfaced_at IS NOT NULL",
        ),
        Index(
            "ix_matches_job_surfaced",
            "job_id",
            text("total_score DESC"),
            postgresql_where="deleted_at IS NULL AND surfaced_at IS NOT NULL",
        ),
        CheckConstraint("vector_score >= 0 AND vector_score <= 1", name="ck_matches_vector_score_range"),
        CheckConstraint("structured_score >= 0 AND structured_score <= 1", name="ck_matches_structured_score_range"),
        CheckConstraint("total_score >= 0 AND total_score <= 1", name="ck_matches_total_score_range"),
        {"schema": "kpa"},
    )
```

`ix_matches_applicant_surfaced` is the P2.3 feed-query index. Defining it now means P2.3 doesn't revisit the migration.

### Scoring modules

**`kpa/scoring/vector.py`:**

```python
import math


def cosine_similarity(a: list[float], b: list[float]) -> float:
    if len(a) != len(b):
        raise ValueError(f"dim mismatch: {len(a)} vs {len(b)}")
    dot = sum(x * y for x, y in zip(a, b))
    na = math.sqrt(sum(x * x for x in a))
    nb = math.sqrt(sum(x * x for x in b))
    if na == 0.0 or nb == 0.0:
        return 0.0
    return dot / (na * nb)


def vector_score(a: list[float], b: list[float]) -> float:
    """Cosine clipped to [0, 1]. Negative similarity → 0 (no anti-signal)."""
    return max(0.0, cosine_similarity(a, b))
```

**`kpa/scoring/structured.py`:**

Three rule fits (`location_fit`, `exp_fit`, `ctc_fit`), each returning `0.0–1.0`. Aggregated by `structured_score(...) -> tuple[float, dict[str, float]]` (unweighted mean + per-rule breakdown).

- **`location_fit`:** 1.0 on any case-insensitive overlap, 1.0 if either side lists "Remote", 0.0 on disjoint sets, 0.5 on either-side-empty (no signal).
- **`exp_fit`:** 1.0 inside the band, linear decay above (slope 1/job_max) and below (slope 1/job_min). 0.5 when applicant years is `None`.
- **`ctc_fit`:** 1.0 in-band or under-band (applicant happy with less). Above band: linear decay to 0 at `1.5 × job_max`. 0.5 when either side has missing data.

Each module is < 100 lines, single responsibility, unit-testable in isolation. The boundary survives future iteration (when `structured.py` grows to 6 rule fits, `match.py` doesn't change).

### Celery tasks

**`kpa/workers/tasks/score_applicant.py`** — task `kpa.score_applicant`, queue `score`.

Two-transaction split (no external API call so no need for the embed worker's three-txn shape):

- **Txn 1:** Load `Applicant` + `ApplicantEmbedding` (joined, both `deleted_at IS NULL`). If missing → log + return. Then `SELECT Job, JobEmbedding` for all `status='open'` jobs that have embeddings (single JOIN'd query).
- **(No DB) compute** all `score_match(...)` results in memory.
- **Txn 2:** UPSERT each row via `pg_insert(Match).on_conflict_do_update(...)`. The `surfaced_at` preservation rule uses `func.coalesce(Match.surfaced_at, sa.case(...))` so the surfaced timestamp is written only when (a) the row is new and crosses threshold, or (b) the row existed with `surfaced_at IS NULL` and the rescore now crosses. Once set, it's preserved on subsequent rescores.

Decorator stack mirrors the embed workers: `bind=True`, `autoretry_for=(TransientScoringError,)`, `max_retries=3`, `acks_late=True`. The inline `_run` thread-hop helper for eager-mode + running-loop is copied (same pattern P1.3 and P2.1 established).

**`kpa/workers/tasks/score_job.py`** — task `kpa.score_job`, queue `score`.

Mirror image: load `Job` + `JobEmbedding`, then SELECT all `(Applicant, ApplicantEmbedding)` pairs (no status filter on applicants), compute in memory, UPSERT each.

### Dispatch wiring

`embed_applicant` Txn 3 (the existing P1.3 worker) gets a `_dispatch_score(applicant_id)` call after the embedding commit. Lazy-import to avoid pulling scoring into the embed module's import path:

```python
def _dispatch_score(applicant_id: UUID) -> None:
    from kpa.workers.tasks.score_applicant import score_applicant
    try:
        score_applicant.delay(str(applicant_id))
    except Exception:
        _log.warning("score.dispatch-failed", applicant_id=str(applicant_id), exc_info=True)
```

Same shape in `embed_job` Txn 3 for `score_job`.

### Settings

`api/src/kpa/settings.py` gets two new fields:

## Data flow

```
embed_applicant Txn 3 COMMIT          embed_job Txn 3 COMMIT
        │                                     │
        ▼                                     ▼
score_applicant.delay(applicant_id)   score_job.delay(job_id)
        │                                     │
        ▼  Txn 1: load applicant + emb,       ▼  Txn 1: load job + emb,
        │  SELECT open jobs with emb           │  SELECT applicants with emb
        │                                     │
        ▼  (no DB) score_match for each        ▼  (no DB) score_match for each
        │                                     │
        ▼  Txn 2: UPSERT N matches             ▼  Txn 2: UPSERT N matches
        │                                     │
        ▼ COMMIT                               ▼ COMMIT
```

Both workers idempotent on rescore. `surfaced_at` preserved via SQL CASE in the UPSERT `set_` clause.

## Testing

- **`tests/unit/test_scoring_structured.py`** — ~22 tests across the three rule fits, table-driven over the cases in the brainstorm (in-band, out-of-band, missing data, remote, etc.).
- **`tests/unit/test_scoring_match.py`** — 6 tests: components recorded, total is weighted sum, threshold boundary, negative cosine clipped, vector_weight=0 path.

### Integration

- **`tests/integration/test_score_applicant_worker.py`** — 8 tests: writes-rows happy path, skips jobs without embeddings, surfaces above threshold, doesn't surface below, idempotent UPSERT, preserves `surfaced_at`, skips deleted applicant, skips when no applicant embedding.
- **`tests/integration/test_score_job_worker.py`** — 5 tests (mirror of the above with axes swapped, drop the surface-related cases since they're already covered by the applicant-side tests).

### What's deliberately not tested in P2.2

- HNSW recall / latency (P2.3 owns it).
- Concurrent score_applicant + score_job for the same (applicant, job) pair (UPSERT makes the race benign).
- Match scoring under model rotation (`model_versions` column captures what was used; cross-version queries are not in scope).
- Per-rule weight tuning (deferred until labeled data exists).

## What's out of scope (intentional)

- **`must_have_skills[]` / `dealbreaker_skills[]` on `Job`** — small follow-up slice with schema + seed-JSON additions.
- **Notice-period rule fit** — needs job-side notice-period bands the schema doesn't capture yet.
- **Per-rule weights as configuration** — deferred until labeled data justifies it.
- **Match explanation text** (P2.4 — blocked on BRD §14 #1; LLM behind an interface).
- **`/v1/feed` and `/v1/jobs/{id}` endpoints** (P2.3).
- **Weekly full re-score Celery beat** — the dispatch chain handles the demo flow; the beat task is operational hardening.
- **Recruiter "candidates for this job" endpoint** — the `ix_matches_job_surfaced` index exists but the HTTP surface is deferred.

## Risks

- **Match quality is unmeasured before labeled data exists.** Mitigation: every match row carries `score_components` and `model_versions` so we can replay arbitrary weight/threshold settings against historical rows without rescoring.
- **Score worker becomes the bottleneck at scale.** Iterating N applicants × M jobs is O(N*M). At MVP scale (100s of applicants × 1000s of jobs) it's fine; at production scale a pre-filter (location prefix, exp band) before vector compute is needed. Mitigation: not a P2.2 problem; revisit at P5 load test.
- **`surfaced_at` semantics surprise users.** A user might expect the feed to drop a match if a rescore lowers the score; we preserve it. Mitigation: documented in CLAUDE.md; if product feedback says otherwise, the rescore path's CASE expression is a one-line change.
- **Vector weight + threshold values are guesses until calibration.** Mitigation: env-driven, no code change to tune; `model_versions` stamp lets us A/B different settings against historical labels.
