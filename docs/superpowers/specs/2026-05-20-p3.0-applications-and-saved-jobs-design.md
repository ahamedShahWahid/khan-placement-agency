# P3.0 ‚Äî Applications + saved jobs ‚Äî design

**Date:** 2026-05-20
**Spec ref:** `IMPLEMENTATION_SPEC.md` ¬ß5 (`applications`, `saved jobs` mentioned in ¬ß1 scope), ¬ß10 (endpoint surface ‚Äî `/v1/jobs/{id}/save`, `/v1/jobs/{id}/apply`, `/v1/applications`, `PATCH /v1/applications/{id}/status`), ¬ß13 P3 ("application tracking; saved jobs").
**Status:** Draft ‚Äî pending approval.

First P3 slice. The applicant can now see surfaced matches with explanations (P2.4); this slice lets them do something with those matches ‚Äî apply, save, withdraw, and review their history. The endpoints are applicant-only; recruiter-side hiring stages are deferred until recruiter HTTP CRUD lands.

## Why this slice

- **Without it the feed is a dead end.** An applicant can browse matches but can't act on them.
- **F2 (Flutter apply + saved + applications UI) consumes these endpoints.** Schemas need to land before frontend can wire up the UI.
- **Two cleanly separable concerns** (applications + saved jobs) share enough conventions to ship together: both are applicant ‚Üî job join tables, both use the same auth ladder, both follow the same partial-UNIQUE idempotency pattern.

## Decisions resolved during brainstorming

| # | Decision | Resolution |
|---|---|---|
| 1 | Application status enum | **Two values: `applied`, `withdrawn`** ‚Äî applicant-controlled only. Recruiter-side stages (`shortlisted`/`interview`/`offered`/`rejected`) deferred to a recruiter slice outside current scope. Single column, not two ‚Äî keeps the schema minimal. |
| 2 | Apply idempotency | **Re-apply returns 200 with the existing row** (no `409 conflict`). Matches the spec ¬ß10 "Idempotent operations" tone. Re-apply against a previously-withdrawn row UPDATEs that same row back to `status='applied'` and refreshes `created_at` ‚Äî preserves row id stability (see Risks for why this matters for cursor pagination). |
| 3 | Save toggle semantics | **POST = save (idempotent), DELETE = unsave.** Re-saving after unsave creates a fresh row. Each save lives in `saved_jobs` with its own soft-delete column. |
| 4 | DELETE-unsaved behavior | **204 No Content** (no-op). UI calls DELETE optimistically without checking state. |
| 5 | Application FK semantics | **`applications.job_id ON DELETE CASCADE`**, `applications.applicant_id ON DELETE CASCADE`. Soft-deletes don't cascade (filtered at query time). Hard delete of an applicant takes their applications; same for hard-deleted jobs (rare). |
| 6 | Source attribution | `source VARCHAR(32) NOT NULL DEFAULT 'feed'`. Recorded on the row at create time. Free-form for now (e.g., `feed`, `direct_link`, `recommended`); promote to enum when values stabilize. |
| 7 | List ordering | Both `/v1/applications` and `/v1/saved` order by `created_at DESC, id DESC`. Cursor pagination identical to `/v1/feed`. |
| 8 | List response shape | **Rich:** each item carries application row + denormalized job summary card + employer summary. Same rationale as the feed (avoid round-trips for the F2 card UX). |
| 9 | Status transition validation | **PATCH only accepts `applied ‚Üí withdrawn`.** Other transitions ‚Üí `400 invalid_transition`. Re-withdraw ‚Üí 200 no-op. |
| 10 | Pre-existing job filter | Apply/save accept only `status='open' AND deleted_at IS NULL` jobs. Closed/soft-deleted job ‚Üí `404 job_not_found`. Same uniform-404 rationale as `/v1/jobs/{id}`. |

## Endpoints

### `POST /v1/jobs/{job_id}/apply`

- Auth: `_require_applicant`. 401/403 ladder identical to feed/job-detail.
- Loads the job: `status='open' AND deleted_at IS NULL`. Missing ‚Üí 404 `job_not_found`.
- Looks up existing live application by `(applicant_id, job_id)`.
  - Existing with `status='applied'` ‚Üí return 200 with the existing row (idempotent).
  - Existing with `status='withdrawn'` ‚Üí INSERT a new row (treated as a re-apply; old withdrawn row stays).
  - None ‚Üí INSERT new row.
- Body: `{"source": "feed"}` (optional; defaults to `feed`).
- Response: `200 ApplicationRead` on existing-match; `201 ApplicationRead` on insert.

### `PATCH /v1/applications/{application_id}`

- Auth: `_require_applicant`.
- Loads the application by id, scoped to current applicant (`applicant_id = current AND deleted_at IS NULL`). Missing or owned-by-another ‚Üí 404 `application_not_found` (uniform).
- Body: `{"status": "withdrawn"}`. Only `applied ‚Üí withdrawn` accepted. Other transitions ‚Üí `400 invalid_transition`. Re-withdraw ‚Üí 200 no-op.
- Response: `200 ApplicationRead`.

### `GET /v1/applications`

- Auth: `_require_applicant`.
- Query: `?limit=20&cursor=...` ‚Äî cursor format `{created_at: ISO8601, application_id: uuid}`.
- Returns rich items: `{application, job, employer}`. Excludes soft-deleted rows; **includes** withdrawn (so the applicant sees their history).
- Order: `created_at DESC, id DESC`.
- ETag: `W/"<sha256(applicant_id + max(updated_at) + count)>"`.

### `POST /v1/jobs/{job_id}/save`

- Auth: `_require_applicant`.
- Job filter: `status='open' AND deleted_at IS NULL`. Missing ‚Üí 404 `job_not_found`.
- Idempotent ‚Äî re-save returns the existing row.
- Response: `200 SavedJobRead` (existing) or `201 SavedJobRead` (new).

### `DELETE /v1/jobs/{job_id}/save`

- Auth: `_require_applicant`.
- Soft-deletes the live row for `(applicant_id, job_id)`. If no live row ‚Üí 204 no-op.
- Response: `204 No Content`.

### `GET /v1/saved`

- Auth: `_require_applicant`.
- Same shape as `/v1/applications`: paginated rich items `{saved_job, job, employer}`. ETag-cached.

## What's out of scope (intentional)

- **Recruiter-side hiring stages** (`shortlisted`, `interview`, `offered`, `rejected`). Deferred.
- **Notifications when an application is created** ‚Äî that wiring lands in P3.1 (notifications outbox dispatch from the apply endpoint).
- **`source` enum tightening** ‚Äî left as free-form `VARCHAR(32)` until values stabilize.
- **Bulk operations** (apply to N jobs in one call).
- **Withdraw ‚Üí re-apply with reason** ‚Äî current flow allows free re-apply; no reason capture.
- **Application status history table** ‚Äî current schema only carries the latest status. If product wants timeline (applied at X, withdrawn at Y, re-applied at Z), add a separate `application_events` table later.

## Risks

- **Partial-UNIQUE + re-apply after withdraw** ‚Äî the index is `(applicant_id, job_id) WHERE deleted_at IS NULL`. Withdrawing doesn't soft-delete the row (it changes `status`); so a re-apply against the same job creates a new row with the same `(applicant_id, job_id)` pair. **The partial-UNIQUE will reject the insert.** Mitigation: on re-apply when the prior row has `status='withdrawn'`, the handler must either (a) soft-delete the old withdrawn row first, or (b) update it back to `applied` with a fresh `created_at`. Approach (b) is simpler and preserves the row id stably ‚Äî go with that. Update the handler logic in the implementation plan accordingly.
- **PATCH idempotency** ‚Äî `applied ‚Üí withdrawn` twice is a no-op (200). `withdrawn ‚Üí applied` is rejected as invalid transition. The handler must distinguish between same-state PATCH and different-target PATCH.
- **`source` is user-supplied** ‚Äî applicant can set arbitrary 32-char strings. No XSS risk (output is always JSON-escaped) but observability dashboards may see junk. Acceptable for MVP; revisit if abuse appears.
- **Cursor identity stability under re-apply-after-withdraw** ‚Äî if approach (b) updates the existing row, the row id stays stable so the cursor format `(created_at, application_id)` doesn't break. If we ever switch to approach (a) we'd need a different cursor scheme.
