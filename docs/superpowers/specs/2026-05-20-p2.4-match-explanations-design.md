# P2.4 — Match explanations (templated fallback) — design

**Date:** 2026-05-20
**Spec ref:** `IMPLEMENTATION_SPEC.md` §6.3 ("Explanation generation"), §7 (LLM provider abstraction), §13 P2 ("why this fits"). BRD §14 Open Decision #1 — LLM provider choice — is **unresolved**, so this slice ships only the templated-fallback path. When the provider is chosen, a follow-up slice swaps the templated impl behind a `MatchExplainer` Protocol.
**Status:** Draft — pending approval.

Fifth and final P2 slice. P2.0–P2.3 produced matches with scores and exposed them via the feed; this slice adds the "why this fits" / "what might not fit" copy spec §6.3 calls out, using deterministic templates derived from `score_components`. The Flutter feed card (F1) will surface the strings directly.

## Why this slice

- **The feed card design space assumes the explanation field exists.** Without it, F1 has empty copy space on every job row, breaking the BRD's "why this fits" promise.
- **The templated fallback is the documented degradation path** when the LLM provider is unavailable (spec §6.3). Shipping it first means the LLM swap later doesn't block on schema changes.
- **Generation cost is negligible.** Sub-millisecond pure-Python string formatting; no need for a new worker or queue.

## Decisions resolved during brainstorming

| # | Decision | Resolution |
|---|---|---|
| 1 | Generation site | **Inline in the score workers' compute step.** No new worker, no new queue. When LLM provider lands, a `MatchExplainer` Protocol routes between templated and LLM impls via env var; the call site doesn't change. |
| 2 | Stored shape | **`{"fit": str, "caveat": str, "generator": "templated", "generator_version": "1"}`** in `matches.explanation JSONB NULL`. `generator` + `generator_version` distinguish rows after a future LLM swap. |
| 3 | Templated rules | **Deterministic priority order** over `(location_fit, exp_fit, ctc_fit, total)`. Five fit-string templates, three caveat templates, English only (i18n deferred). |
| 4 | Surface | **One new field on `MatchRead`** (`routes/feed.py`); both `/v1/feed` and `/v1/jobs/{id}` carry it via the shared model. No route-handler changes. |
| 5 | Nullable | **Yes** — `explanation IS NULL` for old matches written before P2.4. P2.4 score worker fills it on the next rescore; no backfill migration needed for the seeded MVP scale. |
| 6 | i18n / Hindi | **Out of scope.** Templated strings are English. Spec §3.7 lists Hindi as MVP language for the Flutter app — the explanation translation lives on the client side when F1 lands, or as a follow-up if quality matters. |

## Surface

### `kpa/scoring/explain.py`

```python
"""Templated match-explanation generator.

Deterministic priority-order rules over score_components and the surrounding
context. Returns a dict with `fit`, `caveat`, `generator`, `generator_version`
fields for storage on matches.explanation.

When the LLM provider decision (BRD §14 #1) is resolved, a `MatchExplainer`
Protocol will wrap both templated and LLM impls behind a single interface
and the score worker call site stays unchanged.
"""

from __future__ import annotations

from decimal import Decimal


GENERATOR = "templated"
GENERATOR_VERSION = "1"


def templated_explanation(
    *,
    components: dict[str, float],
    vector: float,
    structured: float,
    total: float,
    threshold: float,
    job_title: str,
    job_locations: list[str],
    job_min_exp_years: int,
    job_max_exp_years: int,
    job_ctc_max: Decimal | None,
    employer_name: str,
    applicant_expected_ctc: Decimal | None,
    applicant_locations: list[str],
) -> dict[str, str]:
    loc = components.get("location", 0.5)
    exp = components.get("exp", 0.5)
    ctc = components.get("ctc", 0.5)

    overlap_loc = _location_for_message(job_locations, applicant_locations)
    remote = "remote" in {x.strip().lower() for x in (job_locations + applicant_locations)}

    # --- fit string ---
    if total < threshold:
        fit = "Lower-confidence match — surfaced for breadth."
    elif loc >= 0.9 and exp >= 0.9 and ctc >= 0.9:
        fit = f"Strong match: {job_title} in {overlap_loc} aligns with your experience level."
    elif loc >= 0.9 and exp >= 0.9:
        fit = f"Good location and seniority fit for {job_title} in {overlap_loc}."
    elif remote:
        fit = f"Remote-friendly match for {job_title} at {employer_name}."
    else:
        fit = f"Possible match for {job_title} based on your skill profile."

    # --- caveat string (first weakness wins) ---
    if exp < 0.6:
        caveat = f"Experience band: this role asks for {job_min_exp_years}–{job_max_exp_years} years."
    elif ctc < 0.6 and applicant_expected_ctc is not None and job_ctc_max is not None and applicant_expected_ctc > job_ctc_max:
        caveat = "Compensation may be below your expectation."
    elif loc == 0.0:
        caveat = "Location mismatch — neither side lists overlap or remote."
    else:
        caveat = ""

    return {
        "fit": fit,
        "caveat": caveat,
        "generator": GENERATOR,
        "generator_version": GENERATOR_VERSION,
    }


def _location_for_message(
    job_locs: list[str], applicant_locs: list[str]
) -> str:
    """Return the first overlapping location for display, or the first job location.

    Case-insensitive match, but preserves the casing from the job side
    (typically the recruiter's canonical spelling)."""
    if not job_locs:
        return "your preferred location"
    a_lower = {x.strip().lower() for x in applicant_locs}
    for loc in job_locs:
        if loc.strip().lower() in a_lower:
            return loc
    return job_locs[0]
```

The applicant-side already JOINs Job + JobEmbedding; extend with `JOIN Employer ON Employer.id = Job.employer_id AND Employer.deleted_at IS NULL` and pull `Employer.name` into the per-job tuple. The job-side already has employer access via `score_job_async`'s join — just add it to the loaded tuple shape.

## What's out of scope (intentional)

- **LLM-generated explanations.** Blocked on BRD §14 #1.
- **`MatchExplainer` Protocol abstraction.** Will appear when the LLM provider lands. Defining the Protocol now without two impls is YAGNI.
- **i18n / Hindi translation.** Templated English only.
- **Rate limiting** on explanation generation. Templated path has no external calls.
- **Caching of explanations** beyond the DB column. LLM path will need cache; templated doesn't.
- **Per-applicant explanation customization** (e.g., tone, length). The wire shape carries enough metadata for the client to render; styling is the frontend's call.
- **Backfill migration for existing rows.** New explanations land on the next rescore; null is fine until then.

## Risks

- **Templated strings drift toward sounding robotic.** Mitigation: this is the documented fallback when LLM is unavailable. The bar is "informative", not "delightful". When LLM arrives, this entire module gets shadowed.
- **Template references job_title which may contain special characters.** Mitigation: the templates are f-strings with values that come from `Job.title` (recruiter-set or seed-fixture). No HTML, no markdown — the strings are rendered as plain text in the Flutter card.
- **Caveat priority order might surprise users when multiple weaknesses exist.** Mitigation: order documented in the design and code comment. If user feedback says otherwise, the priority list is a one-function change.
- **`generator_version: "1"` will need bumping if the templates change semantically.** Mitigation: documented in the docstring; reviewers should flag template edits as version-bump candidates.
